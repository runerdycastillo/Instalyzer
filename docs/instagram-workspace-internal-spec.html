Internal Product Spec

Instagram Workspace Internal Spec

Build a dataset-based Instagram SaaS where users upload their Instagram export once, create reusable datasets, and run multiple tools against those datasets.

Core model: Instagram export -> processed dataset -> dataset workspace -> multiple tools -> results

Key principle: Upload once. Use many times.

Product Positioning

Use workspace or platform for the product identity. Use analyzer as a capability or tool type inside the platform.

Working positioning: Upload your Instagram export once, create a reusable dataset, and run AI-powered tools to analyze and improve your account.

Core Product Model

The primary object is the Dataset. A dataset is a processed Instagram export that stores normalized account data, metadata, and tool-ready records for repeated use across multiple tools.

  • import source context
  • normalized follower/following records
  • metadata
  • dataset name
  • notes
  • creation timestamps
  • tool-compatible derived records
  • saved outputs/history later

Core Workflow

  1. User lands on homepage
  2. User uploads Instagram export
  3. System validates and processes files
  4. User reviews upload summary
  5. User names dataset and adds optional notes
  6. System creates dataset
  7. User enters dataset workspace
  8. User selects a tool
  9. Tool runs against dataset
  10. User reviews results

Dataset Workspace Metrics

Base workspace metrics should stay factual and directly supported by the data.

  • follower count
  • following count
  • mutual count
  • not-following-back count
  • imported file count
  • import date
  • dataset created date

Advanced inferred insights such as engagement rate, active hour, or account activity patterns should be tool outputs unless the export clearly supports them as stable dataset-level metrics.

Tool System

One dataset can power multiple tools.

MVP Tool

Not Following Back

  • compare following vs followers
  • identify users who do not follow back
  • review/manage results
  • copy/export results
  • support workflow handling

Near-Term Tools

  • Mutuals / relationship analysis
  • Ghost follower analysis
  • Audience/activity insights

Later Tools

  • Posting-time recommendations
  • AI growth recommendations
  • exportable creator/account reports

Navigation Model

Top-level navigation should include Home, Login / Signup, App, Datasets, Tools, and Account / Billing later.

In-app flow should center on Datasets list, Dataset workspace, Tool page, and Results/output view.

Current Codebase Fit

What already aligns: homepage direction, dataset-first onboarding, upload/parsing flow, Tool 1, and local dataset handoff.

What is temporary: static front-end architecture, prototype-first file structure, limited persistence, and no real SaaS backend/auth model yet.

Technical Direction

The current static HTML/CSS/JavaScript stack is good for prototype and workflow validation, but it is not the ideal final architecture for a real SaaS.

Best long-term fit: Next.js / React plus a backend for users, datasets, jobs, and tool outputs.

Do not rewrite immediately just for fashion. Rewrite when the page structure is stable, the dataset model is clearer, the second and third tools are defined, and auth/account architecture becomes necessary.
